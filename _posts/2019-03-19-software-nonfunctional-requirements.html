Overview

This is a set of non-functional requirements I developed along side the lead engineer on the product team we were starting up from scratch. This set of requirements were to serve as best practices, and were shared with all developers and team members when they came onto the team. These guidelines were frequently referred to during retrospectives, definitions of done conversations, etc.

Requirements

User understands the state of the system at all times

  • The user should know their actions are being interpreted by the system
  • The user should know if something is loading as a result of their action
  • The user should be prompted by the system if something requires their attention or action

Processing & Response times

  • Pages and data should load w/in 2 seconds
  • Components should update w/in 2 seconds after the user makes a request
  • Provide an indication the system is working

Do not refresh the page for new or updated content

  • Components and their data should update independent of the page
  • Do not perform page refreshes for new data unless the user is navigating to a logically different area/page of the site

User has control of how they navigate through the system

  • Do not move the user to another page unless they choose to do so
  • Do not place the user in obscure locations in the system
  • Warn the user if they are about to leave their product, and give them the option to abort or proceed

Accessibility

  • Markup is to be well-formed to allow the content to be easily understood. Where at all possible, the content should be available when styles are not enabled.
  • Colors and contrast should be tested to make sure they are visible to anyone with most sight disabilities.
  • Important content should be immediately visible in the UI. Do not reveal important content via gestures, mouse hovers, or clicks.

Front End Accessibility

UX, UI, and developers are able to audit front end pages and understand, at least on a basic level, what the code is doing.

Data-Driven

  • UI consumes JSON data, and should not use HTML driven techniques such as Ajax
  • Data updates are real-time asynchronous using a technology such as Web Sockets (this is an example only)

Security/Roles

  • The application must support multiple roles associated with the logged in user
  • The application must allow UI elements to be shown to the user that are relevant to the user based the user's roles

Audit/Hipaa

The application must be capable to reporting on a user's data views compatible with HIPAA requirements

Localization

Must use localization techniques that will allow for strings and other elements (such as dates & money) to be localized to match the user's preferred language and culture.

Availability

  • The application must be capable of being run in a web farm, load balanced environment
  • The application must be capable of logging errors and internal information to aid in diagnosing issues
